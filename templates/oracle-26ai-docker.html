004 — Oracle 26ai Free on Docker

Running a full Oracle 26ai database locally on macOS with Colima and Docker — and wiring it up with SQLcl Connection Manager

1. Why Run Oracle 26ai Locally

Cloud databases are great for production. For development, iteration, and experimentation they come with friction — latency, cost, shared environments, and the overhead of managing cloud credentials just to run a quick test. A local Oracle instance changes that entirely.

Oracle Database 26ai Free is the full Oracle database engine, available as a container image, with no license cost and no feature restrictions that matter for development. You get the complete SQL engine, PL/SQL, JSON, and — critically — native vector search with the VECTOR datatype and HNSW indexing. All of it running on your laptop, offline if needed, restarting in seconds.

This article walks through getting Oracle 26ai Free running on macOS using Colima and Docker, then connecting to it cleanly using SQLcl Connection Manager — so your local instance is one command away from any terminal session.

2. Prerequisites

You will need the following tools installed before starting:

ToolPurposeInstall
ColimaContainer runtime for macOS (Docker Desktop alternative)brew install colima
Docker CLIContainer management commandsbrew install docker
SQLclOracle command-line SQL clientoracle.com/sqlcl
Oracle accountRequired to pull from Oracle Container Registryprofile.oracle.com

If you do not have Colima and Docker installed yet:

brew install colima docker

3. Start Colima with Enough Resources

Oracle Database needs meaningful memory and disk to start reliably. The default Colima VM is undersized for it. Start Colima with at least 4 GB of RAM and 60 GB of disk:

colima start --cpu 2 --memory 4 --disk 60
Apple Silicon (M-series) note: Oracle's container image is built for linux/amd64. Colima handles this transparently on Apple Silicon via Rosetta 2 emulation — no extra flags needed. Startup will be slower than on Intel, but it works reliably. If you have 16 GB of RAM, bumping --memory to 6 makes the database noticeably more responsive.

To check that Colima is running:

colima status

4. Log in to Oracle Container Registry

Oracle's container images require accepting a license agreement before pulling. Go to container-registry.oracle.com, sign in with your Oracle account, navigate to Database → Free, and accept the license. You only need to do this once.

Then authenticate Docker with the registry:

docker login container-registry.oracle.com

Enter your Oracle account email and password when prompted.

5. Run the Oracle 26ai Free Container

Pull and start the container in a single command. The ORACLE_PWD environment variable sets the password for the SYS, SYSTEM, and PDBADMIN accounts. The volume mount keeps your data across container restarts:

docker run -d \
  --name oracle26ai \
  -p 1521:1521 \
  -e ORACLE_PWD=YourStrongPassword \
  -v oracle26ai-data:/opt/oracle/oradata \
  container-registry.oracle.com/database/free:latest
FlagWhat it does
-dRun in background (detached)
--name oracle26aiGive the container a memorable name
-p 1521:1521Expose the Oracle listener port to localhost
-e ORACLE_PWDSet SYS / SYSTEM / PDBADMIN password
-v oracle26ai-data:...Persist database files in a named Docker volume

6. Wait for the Database to Be Ready

Oracle takes a few minutes to initialise on first run — it is creating the CDB, the PDB, and all the system tablespaces. Follow the logs:

docker logs -f oracle26ai

Wait until you see this line — it means the database is fully up and accepting connections:

#########################
DATABASE IS READY TO USE!
#########################

On Apple Silicon with 4 GB allocated, expect 3–5 minutes on first startup. Subsequent starts (with the volume already initialised) are much faster — typically under a minute.

Service names to know: The container runs two database services. FREE is the Container Database (CDB) — the root. FREEPDB1 is the Pluggable Database (PDB) where you do your actual work. Always connect to FREEPDB1 for development.

7. Connect with SQLcl and Save with Connection Manager

Once the database is ready, open a new terminal and connect. If this is your first time connecting, do it manually to verify everything works:

sql system/YourStrongPassword@localhost:1521/FREEPDB1

Once confirmed, save it as a named Connection Manager entry so you never have to type the connection string again:

-- Save connection with stored password
conn -save dev26ai -savepwd system/YourStrongPassword@localhost:1521/FREEPDB1

-- Verify it works
cm test dev26ai

-- From now on, connect with a single command
sql -name dev26ai

The -savepwd flag stores the password encrypted in the Oracle DBTools connection store at ~/.dbtools/connections/. You will not be prompted for it again. For a full Connection Manager reference, see article 003.

8. Verify the 26ai Features Are Available

Connect to your instance and run a few quick checks to confirm you have a working Oracle 26ai environment:

sql -name dev26ai

-- Check database version
SELECT banner FROM v$version;

-- Confirm VECTOR datatype is available
SELECT data_type FROM all_types WHERE type_name = 'VECTOR';

-- Check vector memory configuration
SELECT name, value FROM v$parameter WHERE name = 'vector_memory_size';

-- Run a minimal vector test
SELECT vector('[1.0, 2.0, 3.0]', 3, FLOAT32) FROM dual;

If the VECTOR type query returns a row and the vector expression evaluates without error, your Oracle 26ai instance is fully operational and ready for vector search workloads.

9. Day-to-Day Usage

Once everything is set up, your daily workflow is minimal:

# Start Colima (if not already running)
colima start

# Start the database container
docker start oracle26ai

# Connect immediately
sql -name dev26ai

# Stop when done (preserves all data in the volume)
docker stop oracle26ai
Tip: Auto-start on Colima launch — If you work with Oracle daily, add docker start oracle26ai to a shell alias or a Colima startup script so the container comes up automatically when you start your day.

Useful container management commands

# Check container status
docker ps -a --filter name=oracle26ai

# Follow logs (useful when troubleshooting startup)
docker logs -f oracle26ai

# Check resource usage
docker stats oracle26ai

# Completely reset the database (destructive — deletes all data)
docker rm -f oracle26ai
docker volume rm oracle26ai-data

10. Troubleshooting

Container exits immediately after start

Almost always a memory issue. Check that Colima was started with at least 4 GB (colima status) and that no other memory-heavy processes are competing. Run docker logs oracle26ai immediately after the exit to see the error.

ORA-12541: No listener

The database is still starting up. Wait for the "DATABASE IS READY TO USE!" log line before attempting to connect. If the message never appears, check docker logs oracle26ai for errors.

Port 1521 already in use

Another Oracle process or listener is already bound to port 1521 on your machine. Either stop the conflicting process or map the container to a different host port:

# Use port 1522 on the host instead
docker run -d --name oracle26ai -p 1522:1521 ...

# Then connect on the alternate port
conn -save dev26ai -savepwd system/YourStrongPassword@localhost:1522/FREEPDB1

Image pull fails with authentication error

Make sure you have accepted the license on container-registry.oracle.com and that docker login container-registry.oracle.com completed successfully before pulling.
