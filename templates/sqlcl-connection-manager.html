1. Why Connection Manager Matters in Oracle 26ai Workflows

If you are a developer or DBA working with Oracle Database 26ai, your day likely spans multiple environments: a local containerised 26ai instance running on Docker or Colima, a cloud-hosted Pluggable Database (PDB) on OCI, an Autonomous Database (ADB) accessed via wallet, and possibly a dedicated vector store for AI pipelines. Manually typing JDBC connection strings, TNS aliases, and wallet paths every single session is not only tedious — it introduces connection errors that are hard to debug under pressure.

SQLcl's Connection Manager (CM) solves this completely. It acts as a named connection registry that persists across SQLcl sessions, integrates directly with the -name flag at startup, and — critically for 26ai automation — satisfies the -savepwd requirement needed by the SQLcl MCP server to enable passwordless, automated database access from Claude Desktop and other AI tools.

🔑 Key insight for 26ai AI tooling The SQLcl MCP server (oracle.sqldeveloper.mcp) requires a saved connection with stored password (-savepwd) to operate without interactive prompts. This means properly configured Connection Manager entries are a prerequisite for any Oracle 26ai AI Optimizer, RAG pipeline, or MCP-based automation workflow.

2. How Connection Manager Works Under the Hood

Connection Manager stores named connections in a JSON file located in the SQLcl user configuration directory. On Linux and macOS this is typically ~/.sqlcl/connections.json. On Windows it sits under %APPDATA%\sqlcl\. The file is not encrypted but passwords — when saved with -savepwd — are obfuscated using a local keystore mechanism tied to your OS user account.

Connection storage location

# Linux / macOS
~/.sqlcl/connections.json

# Windows
%APPDATA%\sqlcl\connections.json

# View raw JSON (useful for debugging or backup/restore)
cat ~/.sqlcl/connections.json | python3 -m json.tool

Each entry in the JSON contains the connection name, connection string, and a flag indicating whether a password is stored. The actual credential is held in a local Java KeyStore (JKS) file alongside the JSON. This design means that connections.json can be safely committed to a dotfiles repository without exposing passwords — but the keystore cannot be shared across machines.

💡 Tip: Backing up connections across machines Export connection strings (without passwords) from connections.json into a version-controlled dotfile. On a new machine, re-import and re-save passwords with -savepwd. This gives you portable, reproducible connection setup without credential exposure.

3. Core Commands — Complete Reference

The following table covers every Connection Manager command you need as a 26ai developer or DBA.

Command Description Example
conn -save <name> -savepwd <connstr> Save a new named connection with stored password conn -save dev26ai -savepwd system/pass@localhost:1521/freepdb1
conn -save <name> -replace -savepwd <connstr> Update (replace) an existing named connection conn -save dev26ai -replace -savepwd system/newpass@localhost:1521/freepdb1
sql -name <name> Connect to a saved connection by name at startup sql -name dev26ai
cm list List all saved connection names cm list
cm show <name> Show details of a specific connection (no password) cm show dev26ai
cm test <name> Test connectivity for a named connection cm test adb_dev
cm delete -conn <name> Remove a named connection cm delete -conn oke_prod
cm export > <file> Export connection names/strings (no passwords) to JSON cm export > ~/dotfiles/oracle_connections.json
cm import <file> Import connections from a JSON file cm import ~/dotfiles/oracle_connections.json

4. Saving Connections — All Connection Types

4.1 Local PDB (Development)

The most common pattern for 26ai local development against a containerised or on-premises PDB:

-- Save a local PDB connection with password
conn -save dev26ai -savepwd system/Oracle123@localhost:1521/freepdb1

-- Verify immediately
cm test dev26ai

4.2 Oracle Autonomous Database (Wallet-based)

Autonomous Database connections require a wallet. SQLcl supports the wallet either via TNS_ADMIN environment variable or inline within the connection string:

-- Option A: TNS_ADMIN environment variable (preferred for ADB)
export TNS_ADMIN=/home/oracle/wallets/adb_dev
conn -save adb_dev -savepwd admin/MySecurePass1@myatp_high

-- Option B: Inline wallet reference
conn -save adb_dev -savepwd \
  'admin/MySecurePass1@(description=(retry_count=20)(retry_delay=3)
   (address=(protocol=tcps)(port=1522)(host=adb.eu-frankfurt-1.oraclecloud.com))
   (connect_data=(service_name=<your_ocid>_high.adb.oraclecloud.com))
   (security=(ssl_server_dn_match=yes)(my_wallet_directory=/wallets/adb_dev)))'

-- Test the ADB connection
cm test adb_dev
⚠️ ADB wallet note The wallet directory must contain cwallet.sso, ewallet.p12, tnsnames.ora, and sqlnet.ora. When downloaded from OCI Console, extract the .zip directly into your wallet directory. Do not rename any files. The -savepwd flag is mandatory for MCP server usage.

4.3 OKE / Kubernetes-hosted PDB (SSH Tunnel or NodePort)

When running Oracle 26ai on Oracle Kubernetes Engine (OKE) with a service exposed on a NodePort or via kubectl port-forward:

-- Assuming kubectl port-forward svc/oracle-db 1521:1521 is active
conn -save oke_prod -savepwd system/Oracle123@127.0.0.1:1521/FREEPDB1

-- Or for a NodePort on the OKE worker node IP
conn -save oke_prod -savepwd system/Oracle123@<node_ip>:30521/FREEPDB1

4.4 Oracle 23ai / 26ai with TNS Alias

If you maintain a tnsnames.ora file, you can save connections using TNS aliases directly. Ideal for teams sharing a centralised tnsnames.ora:

-- Assuming TNS_ADMIN is set and tnsnames.ora defines 'db26ai_dev'
conn -save db26ai_dev -savepwd hr/Oracle123@db26ai_dev

4.5 Proxy (Application User) Connections

Oracle 26ai supports proxy authentication — useful when an app schema user connects through a proxy account. Connection Manager supports this natively:

-- Proxy connection: connect as appuser through proxyaccount
conn -save app_proxy -savepwd appuser[proxyaccount]/ProxyPass@localhost:1521/freepdb1

5. Managing Connections — Inspect, Test, Update, Remove

5.1 Listing and Inspecting

-- List all saved connection names
cm list

-- Example output:
.
├── adb_dev
├── db26ai_dev
├── dev26ai
└── oke_prod

-- Show details of a specific connection
cm show dev26ai

-- Example output:
Name      : dev26ai
Username  : system
ConnString: localhost:1521/freepdb1
SavePwd   : true

5.2 Testing Connectivity

Connection testing is particularly valuable in Oracle 26ai pipelines where you need to validate connectivity before triggering a migration, vector ingestion job, or AI pipeline run:

-- Test a single connection
cm test dev26ai

-- Batch-test all connections using shell loop
for conn in dev26ai adb_dev oke_prod db26ai_dev; do
  echo -n "Testing $conn ... "
  sql -name $conn -S /nolog << EOF
  cm test $conn
  EXIT
EOF
done

5.3 Updating Credentials

The -replace flag updates an existing entry without needing to delete and recreate it:

-- Update password after rotation
conn -save dev26ai -replace -savepwd system/NewPassword123@localhost:1521/freepdb1

-- Confirm the update worked
cm test dev26ai

5.4 Deleting Connections

-- Remove a named connection
cm delete -conn oke_prod

-- Verify removal
cm list

6. Connection Manager and the SQLcl MCP Server

This is where Connection Manager becomes not just convenient but architecturally essential for Oracle 26ai AI workflows. The SQLcl MCP server — part of the oracle.sqldeveloper.mcp extension — exposes Oracle database tools (query execution, object listing, DDL generation) to AI assistants like Claude Desktop via the Model Context Protocol.

The MCP server launches SQLcl as a subprocess using stdio transport. Because there is no interactive terminal, it cannot prompt for passwords. The -name flag combined with a -savepwd connection is the only supported authentication path.

Claude Desktop MCP configuration

A typical claude_desktop_config.json entry for the SQLcl MCP server targeting an Oracle 26ai instance:

// claude_desktop_config.json
{
  "mcpServers": {
    "oracle-db": {
      "command": "/opt/sqlcl/bin/sql",
      "args": ["-name", "dev26ai"]
    },
    "oracle-adb": {
      "command": "/opt/sqlcl/bin/sql",
      "args": ["-name", "adb_dev"]
    }
  }
}
🔒 Security best practice Never put credentials directly in claude_desktop_config.json or in MCP server command arguments. Using -name keeps credentials out of process argument lists (visible via ps aux) and out of version-controlled config files. The Connection Manager keystore is the right credential boundary.

Using SQLcl MCP with multiple databases

You can register multiple MCP servers in Claude Desktop, simultaneously exposing your local 26ai dev database, an ADB instance, and a production PDB — each as a separate MCP server:

// Multi-DB claude_desktop_config.json
{
  "mcpServers": {
    "oracle-dev": {
      "command": "/opt/sqlcl/bin/sql",
      "args": ["-name", "dev26ai"]
    },
    "oracle-adb-prod": {
      "command": "/opt/sqlcl/bin/sql",
      "args": ["-name", "adb_dev"]
    },
    "oracle-vectorstore": {
      "command": "/opt/sqlcl/bin/sql",
      "args": ["-name", "db26ai_dev"]
    }
  }
}

7. Vector Store Connections for Oracle 26ai AI Pipelines

Oracle Database 26ai elevates vector search to a first-class citizen with HNSW and IVF indexing, the VECTOR datatype, and the DBMS_VECTOR package. When you build RAG pipelines — whether using LangChain, Spring AI, or the Oracle AI Optimizer — your embedding ingestion and retrieval scripts connect to the vector store database repeatedly.

Recommended naming convention for AI workloads

-- Vector store database (26ai with HNSW indexes)
conn -save vs_dev -savepwd vector_admin/VecPass123@localhost:1521/freepdb1

-- Oracle AI Optimizer metadata schema
conn -save ai_opt -savepwd ai_opt_user/OptPass123@localhost:1521/freepdb1

-- Production ADB vector store
conn -save vs_prod -savepwd admin/ProdPass123@myadb_medium

-- Verify vector memory configuration after connecting
sql -name vs_dev
SQL> SELECT value FROM v$parameter WHERE name = 'vector_memory_size';
SQL> SELECT index_name, index_type FROM user_indexes WHERE index_type LIKE '%VECTOR%';

Python integration via subprocess

Oracle AI Optimizer scripts and LangChain-based embedding pipelines often invoke SQLcl programmatically. Connection Manager names make this clean and portable:

import subprocess
import sys

def run_sqlcl(conn_name: str, sql: str) -> str:
    """Execute SQL via SQLcl named connection, return stdout."""
    result = subprocess.run(
        ['/opt/sqlcl/bin/sql', '-name', conn_name, '-S', '/nolog'],
        input=f"{sql}\nEXIT;",
        capture_output=True,
        text=True
    )
    if result.returncode != 0:
        raise RuntimeError(f'SQLcl error: {result.stderr}')
    return result.stdout

# Query vector index status on dev
output = run_sqlcl('vs_dev',
    "SELECT index_name, index_type, status FROM user_indexes"
    " WHERE index_type LIKE '%VECTOR%'")
print(output)

8. Automating Connection Setup — CI/CD and Team Onboarding

In team environments or CI/CD pipelines running Oracle 26ai integration tests, you need a repeatable way to bootstrap Connection Manager entries without manual interaction.

8.1 Headless connection setup script

#!/bin/bash
# setup_connections.sh — run once per environment
# Requires: DB_PASSWORD and ADB_PASSWORD env vars set
set -euo pipefail
SQLCL=/opt/sqlcl/bin/sql

echo "Setting up SQLcl Connection Manager entries..."

# Local 26ai dev PDB
$SQLCL -S /nolog << EOF
conn -save dev26ai -replace -savepwd system/${DB_PASSWORD}@localhost:1521/freepdb1
cm test dev26ai
EXIT
EOF

# Vector store connection
$SQLCL -S /nolog << EOF
conn -save vs_dev -replace -savepwd vector_admin/${DB_PASSWORD}@localhost:1521/freepdb1
cm test vs_dev
EXIT
EOF

# ADB (TNS_ADMIN must be pre-configured)
export TNS_ADMIN=/wallets/adb_dev
$SQLCL -S /nolog << EOF
conn -save adb_dev -replace -savepwd admin/${ADB_PASSWORD}@myatp_high
cm test adb_dev
EXIT
EOF

echo "All connections configured successfully."

8.2 Exporting and importing connections

For team onboarding, export connection names (structure only, no passwords) and share via a repository:

-- Export (run inside SQLcl)
cm export > ~/dotfiles/oracle_connections.json

-- Import on a new machine
cm import ~/dotfiles/oracle_connections.json

-- Then add passwords individually
conn -save dev26ai -replace -savepwd system/MyLocalPwd@localhost:1521/freepdb1
conn -save adb_dev  -replace -savepwd admin/MyAdbPwd@myatp_high

8.3 GitHub Actions workflow example

# .github/workflows/oracle26ai-test.yml
name: Oracle 26ai Integration Tests
on: [push, pull_request]
jobs:
  integration-test:
    runs-on: ubuntu-latest
    services:
      oracle26ai:
        image: container-registry.oracle.com/database/free:latest
        env:
          ORACLE_PWD: ${{ secrets.ORACLE_PWD }}
        ports: ['1521:1521']
    steps:
      - uses: actions/checkout@v4
      - name: Setup SQLcl Connection Manager
        env:
          DB_PASSWORD: ${{ secrets.ORACLE_PWD }}
        run: bash scripts/setup_connections.sh
      - name: Run SQL regression tests
        run: |
          sql -name dev26ai -S @tests/regression_suite.sql

9. Advanced Patterns and Diagnostics

9.1 Environment-specific connection switching

Use shell environment variables to drive connection selection, making scripts portable across dev/staging/prod:

#!/bin/bash
# deploy_schema.sh
ENV=${1:-dev}  # default to dev
case $ENV in
  dev)     CONN=dev26ai  ;;
  staging) CONN=vs_dev   ;;
  prod)    CONN=adb_dev  ;;
  *)       echo "Unknown env: $ENV"; exit 1 ;;
esac

echo "Deploying schema to: $CONN ($ENV)"
sql -name $CONN @schema/deploy.sql

9.2 Diagnosing ECONNREFUSED and wallet errors

When cm test returns an error, the following diagnostic sequence isolates the cause quickly:

-- Step 1: Confirm the listener is reachable
nc -zv localhost 1521

-- Step 2: Check TNS_ADMIN for wallet-based connections
ls -la $TNS_ADMIN
# Must contain: cwallet.sso  ewallet.p12  sqlnet.ora  tnsnames.ora

-- Step 3: Verify the saved connection string
cm show adb_dev

-- Step 4: Test with verbose SQLcl output
sql /nolog
SQL> SET CLOUDCONFIG /path/to/wallet.zip
SQL> conn admin@myatp_high

-- Step 5: Check for keystore corruption — delete and re-save
cm delete -conn adb_dev
conn -save adb_dev -savepwd admin/NewPass@myatp_high

9.3 Multiple SQLcl versions side-by-side

Oracle 26ai development sometimes requires testing against both SQLcl 24.x and SQLcl 25.x. Each version uses the same ~/.sqlcl directory, but you can isolate per version with SQLCL_HOME:

# Isolate connection store for a specific SQLcl version
export SQLCL_HOME=/home/oracle/.sqlcl_v25
/opt/sqlcl_v25/bin/sql -name dev26ai

11. Conclusion

SQLcl Connection Manager is a small feature with outsized impact for Oracle 26ai development workflows. For individual developers, it eliminates the cognitive overhead of managing multiple connection strings across different environments. For DBAs, it provides a consistent, testable connection baseline. For AI and automation workloads — particularly the SQLcl MCP server integration with Claude Desktop and the Oracle AI Optimizer — it is architecturally essential: the -savepwd requirement makes it a hard dependency, not an optional convenience.

As Oracle 26ai continues to expand its AI-native capabilities — vector indexing, SELECT AI, DBMS_VECTOR, and the emerging RAG-over-MCP architecture — having a clean, named connection strategy from day one will save you from an entire class of connectivity debugging that distracts from the actual work.

If you are building on Oracle 26ai today, establish your Connection Manager entries as the very first step. Everything else builds on top of them.
