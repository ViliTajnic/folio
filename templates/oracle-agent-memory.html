006 — Oracle 26ai as Durable Memory for OpenClaw Agents

A local-first case study: using Oracle Database 26ai Free as the durable memory store for OpenClaw agents, while keeping OpenClaw's existing memory_search experience working.

1. The Problem: My Agent Needed a Memory That Survived

When you start experimenting with coding agents, the first few sessions feel magical. The agent can inspect a repo, write code, run commands, and explain what it did. Then you come back the next day and hit the classic problem: the agent only knows what made it into its current context or into whatever memory files the environment already maintains.

For a small test environment, Markdown memory is often enough. It is simple, transparent, and easy to debug with cat, grep, and your editor. But I wanted one more thing: a durable memory store that behaved like application infrastructure, not just a pile of notes.

That led to this experiment: use Oracle Database 26ai Free as the durable memory store for OpenClaw agents, while keeping OpenClaw's existing memory_search experience working.

The important design decision was not to replace OpenClaw on day one. Instead, Oracle becomes the system of record, and a small compatibility layer mirrors memory entries back into Markdown so OpenClaw can still find them immediately.

We will build:

  • a dedicated PDB (OPENCLAW)
  • a dedicated tablespace (OPENCLAW)
  • a single schema owner used by all agents (CLAW)
  • a minimal memory schema (AGENTS, SESSIONS, MEMORIES, MEMORY_EVENTS)
  • a simple full-text search index for keyword lookup with Oracle Text
  • two helper CLIs: oc-mem-add for writing memory and oc-mem-search for retrieving memory

And we will make it feel automatic in OpenClaw by mirroring durable memories into Markdown files such as memory/YYYY-MM-DD.md. The agent can keep using the built-in memory tools while the database quietly becomes the durable source underneath.

2. The Shape of the Solution

The whole experiment can be summarized like this:

Developer or agent
      |
      v
bin/oc-mem-add
      |
      +--> Oracle Database 26ai Free
      |       AGENTS
      |       SESSIONS
      |       MEMORIES
      |       MEMORY_EVENTS
      |
      +--> memory/YYYY-MM-DD.md
              OpenClaw memory_search can index this immediately

For reads there are two paths:

OpenClaw memory_search           bin/oc-mem-search
      |                                |
      v                                v
Markdown mirror                  Oracle Text CONTAINS() over MEMORIES.CONTENT

This is intentionally boring. Boring is good for agent memory. You want every write to be inspectable. You want every row to have a timestamp. You want to know which agent wrote it, what kind of memory it is, and whether it came from a preference, a fact, a summary, or a tool result.

3. Why Oracle Here

Oracle Database 26ai Free is a good fit for this local-first experiment because it gives the memory system a serious backend from the beginning:

  • durable writes
  • transactions and concurrency control
  • a relational schema for long-term memory objects
  • native JSON columns for tags, metadata, and provenance
  • Oracle Text for keyword search
  • a path toward vector search and hybrid retrieval later

Agent memory usually starts simple and then grows in two directions at once. On one side, the data becomes more structured — you want to know whether a memory belongs to a user, a project, a session, or a specific agent, plus importance, confidence, source, tags, and timestamps. On the other side, retrieval becomes more interesting — keyword search is useful, but eventually you also want semantic search, ranking, deduplication, summarization, and retention policies.

A database gives you a stable place to evolve those ideas without constantly changing the agent-facing workflow.

4. Memory Wants to Become Graph, Log, and Search

Where Oracle shines is if you want memory to become more like a knowledge graph + event log + vector search system instead of just text files.

That sounds abstract, but for programmers it becomes concrete very quickly. A Markdown file is excellent for a human-readable note — it answers "What did we write down?" But once agent memory becomes part of a real workflow, the interesting questions are no longer only text questions:

  • Which agent learned this?
  • Which session or task produced it?
  • Is this a user preference, a project fact, a decision, or a temporary observation?
  • Which memory supersedes an older one?
  • Which memories are connected to the same project, repo, feature, customer, or incident?
  • What did the agent believe before it changed its answer?
  • Which memories are important enough to retrieve first?

Memory as a Knowledge Graph

A useful agent memory is not just a list of paragraphs. It is a web of things and relationships:

user        -> prefers     -> Oracle-first local infrastructure
project     -> uses        -> OpenClaw
agent       -> wrote       -> memory 42
memory 51   -> supersedes  -> memory 17
session     -> produced    -> summary memory 88

A future extension could look like this:

create table MEMORY_LINKS (
  from_memory_id number references MEMORIES(memory_id),
  to_memory_id   number references MEMORIES(memory_id),
  relation       varchar2(64) not null,
  created_at     timestamp default systimestamp not null,
  metadata       json,
  primary key (from_memory_id, to_memory_id, relation)
);

You can start with normal tables and SQL, and then move toward property-graph-style thinking without throwing the first version away. Oracle AI Database 26ai includes SQL property graph capabilities, so this is not a separate architectural island.

Memory as an Event Log

The current fact is important, but the change history is often just as important. If we simply edit a text file, we lose the story of what changed. With an event table, we can record that a memory was created, corrected, archived, merged, or superseded.

That is why the prototype already includes MEMORY_EVENTS. Later it can hold events such as:

created
retrieved
updated
archived
superseded
summarized_from_session
merged_with_duplicate
confidence_lowered

This turns memory into an audit trail. A durable event log lets you answer: "Why does the agent believe this?" instead of only "What does the agent believe now?"

Memory as Vector Search

Keyword search is the first useful retrieval layer. But a user may ask about "persistent agent state" while the stored memory says "durable OpenClaw memory." Those phrases are related, but they do not share enough words for plain keyword search to be reliable.

That is where vector search becomes useful. Oracle AI Vector Search provides a VECTOR data type and SQL functions such as VECTOR_DISTANCE. The important part is that the vector lives beside the rest of the memory record:

-- Future shape, once an embedding column exists.
alter table MEMORIES add content_embedding vector;

select memory_id, title, content
from   memories
where  agent_id = :agent_id
  and  scope in ('user', 'project')
  and  archived_at is null
order  by vector_distance(content_embedding, :query_embedding),
          importance desc,
          created_at desc
fetch  first 10 rows only;

The vector is part of the same memory object — that avoids a lot of glue code and a lot of synchronization bugs.

5. Design Goals and Constraints

The goals were deliberately small:

  • Durable memory lives in Oracle.
  • All agents share the same database user (CLAW) — this is a single-user test system.
  • No secrets are stored in git.
  • No passwords are printed in blog snippets.
  • OpenClaw memory search keeps working without a custom backend.

The main constraint is OpenClaw's existing memory model. OpenClaw indexes Markdown files (MEMORY.md, memory/*.md, and nested files under memory/). A full Oracle-backed OpenClaw memory provider would be cleaner long term, but it is a bigger integration project. For this experiment, the pragmatic solution is:

Oracle           = durable source of truth
Markdown mirror  = OpenClaw compatibility layer

6. Prerequisites

You need:

  • Oracle Database 26ai Free running locally as a CDB (see article 004)
  • SQLcl installed, ideally a recent 26.x version
  • a SYSDBA connection profile configured in SQLcl, so you can run sql -name <your_sys_profile>

Optional: SQLcl MCP mode, if you want to expose Oracle operations through MCP tooling. Keep the configuration sanitized in articles, docs, and examples:

7. Step 1 — Create a Dedicated PDB

The first choice is isolation. A dedicated PDB makes the experiment easier to reason about, easier to drop, and easier to back up. Connect as SYS using your SQLcl profile and create the PDB:

-- Run in CDB$ROOT
create pluggable database OPENCLAW
  admin user PDBADMIN identified by "<strong-password>"
  file_name_convert = (
    '<seed-path>/', '<data-path>/OPENCLAW/',
    '<SEED-PATH>/', '<data-path>/OPENCLAW/'
  );

alter pluggable database OPENCLAW open;
alter pluggable database OPENCLAW save state;
file_name_convert is environment-specific; use whatever matches your Oracle Free layout. save state ensures the PDB reopens on restart.

8. Step 2 — Create the Tablespace and Schema Owner

Switch into the PDB and create a dedicated tablespace and one schema owner for the memory tables:

alter session set container = OPENCLAW;

create tablespace OPENCLAW
  datafile '<data-path>/OPENCLAW/openclaw01.dbf'
  size 256m
  autoextend on next 128m maxsize 4096m;

create user CLAW identified by "<strong-password>"
  default tablespace OPENCLAW
  temporary tablespace TEMP
  quota unlimited on OPENCLAW;

grant create session to CLAW;
grant db_developer_role to CLAW;

The mental model is simple:

OPENCLAW PDB
  OPENCLAW tablespace
    CLAW schema
      memory tables

9. Step 3 — Create the Minimal Durable-Memory Schema

Connect as CLAW and create the four tables:

TablePurpose
AGENTSOne row per agent identity.
SESSIONSOptional session tracking — useful when memory comes from a chat or task run.
MEMORIESThe main durable memory table.
MEMORY_EVENTSAudit-style events around memory creation, updates, merges, or archival.
 
create table AGENTS (
  agent_id       varchar2(128) primary key,
  display_name   varchar2(256),
  created_at     timestamp default systimestamp not null,
  metadata       json
);

create table SESSIONS (
  session_id     varchar2(256) primary key,
  agent_id       varchar2(128) references AGENTS(agent_id),
  channel        varchar2(128),
  created_at     timestamp default systimestamp not null,
  last_seen_at   timestamp,
  metadata       json
);

create table MEMORIES (
  memory_id      number generated always as identity primary key,
  agent_id       varchar2(128) references AGENTS(agent_id),
  session_id     varchar2(256),
  scope          varchar2(64) default 'user' not null,
  kind           varchar2(64) default 'fact' not null,
  title          varchar2(512),
  content        clob not null,
  importance     number(3) default 50 check (importance between 0 and 100),
  confidence     number(5,4) default 1 check (confidence between 0 and 1),
  source         varchar2(512),
  tags           json,
  metadata       json,
  created_at     timestamp default systimestamp not null,
  updated_at     timestamp default systimestamp not null,
  archived_at    timestamp
);

create table MEMORY_EVENTS (
  event_id       number generated always as identity primary key,
  memory_id      number references MEMORIES(memory_id),
  event_type     varchar2(64) not null,
  actor_agent_id varchar2(128),
  payload        json,
  created_at     timestamp default systimestamp not null
);

create index MEMORIES_AGENT_SCOPE_IX on MEMORIES(agent_id, scope, archived_at);

insert into AGENTS(agent_id, display_name)
values ('main', 'OpenClaw main agent');
commit;

This schema is intentionally small, but it already gives us useful handles:

  • scope can separate user memory, project memory, system memory, or team memory.
  • kind can distinguish facts, preferences, summaries, decisions, and instructions.
  • importance gives future ranking logic a starting point.
  • confidence lets the agent store uncertainty instead of pretending every memory is perfect.
  • tags and metadata keep the core table flexible without turning every experiment into a migration.
  • archived_at lets us hide old or superseded memories without deleting them immediately.

10. Step 4 — Add Keyword Search with Oracle Text

For predictable CONTAINS() behavior, use a classic Oracle Text CONTEXT index over the content column:

create index MEMORIES_CONTENT_CTX on MEMORIES(content)
  indextype is ctxsys.context
  parameters ('sync (on commit)');

select memory_id, title
from   memories
where  contains(content, 'oracle') > 0
order  by memory_id desc;

For a local test environment, sync (on commit) is convenient because a memory becomes searchable right after the helper script commits it. For heavier write volume, you would revisit the indexing strategy and maintenance settings.

11. Step 5 — Wire It Into OpenClaw Without Replacing It

OpenClaw already knows how to index memory Markdown files. That is the feature we reuse. The helper script does two writes:

  1. insert the memory into Oracle
  2. append a readable entry to memory/YYYY-MM-DD.md

That makes Oracle durable and OpenClaw compatible at the same time.

Local secret file

Create secrets/openclaw_oracle.env (do not commit this file):

export OPENCLAW_ORACLE_CONNECT='claw/<password>@localhost:1521/OPENCLAW'

Helper scripts

The repository has two small command-line helpers — bin/oc-mem-add writes to Oracle and mirrors to Markdown, while bin/oc-mem-search searches Oracle via CONTAINS(). Example usage:

source secrets/openclaw_oracle.env

./bin/oc-mem-add \
  --agent main \
  --scope user \
  --kind preference \
  --title "DB preference" \
  --content "Single-user system: all agents share the CLAW schema. Oracle is the durable memory source; Markdown is the OpenClaw compatibility mirror." \
  --tags-json '["preference","oracle","openclaw","memory"]'

./bin/oc-mem-search --q "Oracle durable memory" --limit 5

The Markdown mirror should be plain enough that a human can read it and a search index can chunk it:

## 2026-05-04 10:15:32 - DB preference

- memory_id: 42
- agent: main
- scope: user
- kind: preference
- tags: preference, oracle, openclaw, memory

Single-user system: all agents share the CLAW schema. Oracle is
the durable memory source; Markdown is the OpenClaw compatibility mirror.

12. Helper Script Behavior

How oc-mem-add should behave

  1. read OPENCLAW_ORACLE_CONNECT from the environment
  2. validate required arguments (--agent, --title, --content)
  3. pass values into SQLcl using binds
  4. run oracle/mem_add.sql
  5. capture the new memory_id
  6. append a Markdown entry to memory/YYYY-MM-DD.md
  7. print a small success message

If the Oracle insert fails, the helper should not append the Markdown mirror. Oracle is the source of truth — the mirror should only reflect committed database writes.

How oc-mem-search should behave

  1. read OPENCLAW_ORACLE_CONNECT from the environment
  2. accept a search query with --q
  3. apply a sensible default limit
  4. run oracle/mem_search.sql
  5. print compact results that include memory_id, title, created_at, and a short content preview

During development, this is very useful because you can answer: did the write reach Oracle? Did the text index pick it up? Is the memory archived? Did the Markdown mirror contain the same content?

13. Implementation Notes — What Bit Us

1. SQLcl, JSON columns, and naive SQL can surprise you

When inserting into JSON columns, avoid clever SQL wrappers that make the statement harder to bind and debug. What worked reliably:

  • bind JSON strings into :tags_json and :meta_json
  • insert them directly into JSON-typed columns in PL/SQL
  • use RETURNING ... INTO to get the new memory_id

The boring version is easier to troubleshoot. That matters when the caller is not always a human — agents are very good at calling tools repeatedly, so the tool should fail clearly and predictably.

2. Prefer a CONTEXT index for CONTAINS()

Oracle has multiple text and search options. For this first version, the goal is not to design the perfect retrieval stack — the goal is to make keyword lookup reliable. For standard CONTAINS() text behavior, a CONTEXT index is the simple choice.

3. Keep the Markdown mirror disposable

The mirror is for compatibility. Treat it as a generated artifact — do not put secrets in it, do not manually edit it, keep the format stable enough for search, and keep the database row as the authority. If the Markdown mirror is deleted, you should be able to regenerate it from Oracle.

4. Store uncertainty instead of hiding it

The confidence column may look unnecessary in a prototype, but it is valuable once agents start summarizing or extracting memories automatically. Direct user preferences can be high confidence; inferences from a conversation should not be stored with the same confidence. A memory system that can represent uncertainty is easier to maintain than one that pretends every extracted fact is equally true.

15. Next Steps — Use More of the Database

Add vector search

Store embeddings next to the memory row, then query by semantic similarity. Combine structured filters (agent_id, scope, kind, archived_at), keyword search over content, vector similarity over embeddings, and ranking based on importance, confidence, and recency.

Real OpenClaw provider

The Markdown mirror is a good compatibility layer, but not the end state. A deeper integration would let OpenClaw query Oracle directly — Oracle becomes both the durable store and the retrieval engine. The mirror can remain useful as an export/debug format.

Retention & summarization

Raw memory grows quickly. A lifecycle pipeline turns episodic chat logs into extracted candidate memories, then summarized semantic memories, then curated long-term memory, then archived or superseded old rows.

16. Operational Safety Checklist

Before letting agents write to durable storage, keep a few rules in place:

  • Keep credentials out of git.
  • Keep credentials out of blog snippets.
  • Use a dedicated PDB or at least a dedicated schema.
  • Start permissive for prototyping, then tighten roles and grants.
  • Treat generated Markdown as public within the workspace.
  • Log memory writes clearly enough that a human can audit them.
  • Prefer archiving over deleting while the system is still young.
  • Make it easy to export or regenerate the Markdown mirror from Oracle.
Agent memory is not just a feature. It is state. Once a system has durable state, debugging and safety become part of the design.

18. Closing Thought

The useful part of this experiment is not that the schema is perfect. It is not. The useful part is the pattern.

Start with the memory flow your agent already understands. Add a durable database behind it. Keep the compatibility layer simple. Make every write observable. Then improve retrieval step by step.

For a programmer, that is a comfortable path: one small tool, one table design, one working feedback loop at a time.
